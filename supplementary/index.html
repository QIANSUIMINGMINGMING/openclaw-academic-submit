Why This Supports The PEV Claim

The two baselines are intentionally strong: one uses a modern one-shot image model, and the other uses an editable vector drawing path. Even so, the published figure is more stable in message structure, label control, and proposal-scale readability because it was not forced through a single generation step.

The point of this supplement is therefore narrow and explicit: it exists only to document the PEV workflow and to show why the final figure was stronger than either single-step baseline used alone.

Self-Contained Reproduction Record

The record below is written for external readers. It avoids internal file paths and gives the exact prompts plus the final PEV workflow record in plain language. 

Common target for all three images
- Show the pain point of parallel research on a shared cluster.
- Use a three-column layout plus a bottom consequence strip.
- Column 1: single agent workflow.
- Column 2: same user launches multiple agents and sees mild contention.
- Column 3: multiple users share one cluster and create hidden occupancy and stronger conflict.
- Bottom strip labels: contention, hidden occupancy, interference, SLO drop.

Prompt A: one-shot OpenRouter image baseline
"Create a single wide academic infographic on a white background for a systems research proposal. Show the pain point of parallel research on a shared cluster. Use a 3-column plus bottom-strip layout. Left column title: 'Single Agent Work' with a simple pipeline local -> build -> upload -> remote -> run. Middle column title: 'Same User Parallel' with one user launching agent A and agent B onto a shared cluster, with mild visible contention. Right column title: 'Multi-User Shared Cluster' with user 1 and user 2 each launching agents into one shared cluster, with hidden occupancy, opaque intent, and stronger conflict. Bottom strip with four tiny labels only: contention, hidden occupancy, interference, SLO drop. Flat vector academic style, icon-first, thin borders, blue/orange/gray/red palette, large readable English labels, no paragraphs, no product names, no watermark, no glossy poster style, no purple, no decorative background."

Prompt B: direct Codex SVG baseline
"Draw a clean academic SVG diagram, not a decorative poster. White background, thin borders, readable English labels, and a 16:9 layout. Make three columns and one bottom strip. Column 1 is 'Single Agent Work' with a local-to-remote pipeline. Column 2 is 'Same User Parallel' with one user launching two agents into one shared cluster, showing mild visible contention. Column 3 is 'Multi-User Shared Cluster' with two users launching agents into one shared cluster, showing hidden occupancy, opaque intent, and stronger conflict. Bottom strip must contain exactly four labels: contention, hidden occupancy, interference, SLO drop. Keep the figure icon-first, editable, and suitable for a proposal PDF."

PEV workflow record for the final published figure
1. Planning
   Write a short message brief before drawing: fix the three-column story, the bottom-strip consequences, the visual hierarchy, and the allowed label budget.
2. Execution
   Draw the figure as a set of controlled vector assets and merged panels instead of relying on one model output. Keep terminology and icon structure aligned with the planning brief.
3. Validation
   Inspect the near-final figure at proposal-PDF scale. Check title readability, label density, panel balance, and whether the visual emphasis still matches the intended message. Revise and rerender until those checks pass.

Interpretation
- The one-shot OpenRouter baseline is strong for rapid visual ideation, but it drifts more easily.
- The direct SVG baseline is strong for exact structure and editability, but it depends heavily on one drafting pass.
- The published figure is stronger because PEV separates planning, execution, and validation instead of asking one step to do everything.
